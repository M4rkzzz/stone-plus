# Stone+ 内置代理能力边界

> 审计日期：2026-07-23
> 对照基线：[GUI.for.SingBox 社区版功能说明](https://gui-for-cores.github.io/guide/gfs/community)
> 适用范围：Stone+ 内置代理（sing-box v1.13.14），不改变既有账号代理、号池代理和 `gateway.outboundNetworkMode`

## 1. 产品定位与审计结论

GUI.for.SingBox 是通用 sing-box 配置前端，目标是让用户编辑尽可能完整的核心配置，因此提供内核管理、入站端口、外部控制器、代理组、路由规则、DNS、规则集和 TUN 高级参数等广泛能力。

Stone+ 内置代理不是通用 sing-box 配置编辑器。它是 Stone+ 出站网络的一条受控路径，并可按用户选择接管操作系统的系统代理或临时 TUN。其首要约束是：

1. 只有核心、mixed 入口、内部控制器和系统代理/TUN 都经主进程验证后，页面和新请求才能显示为“已接管”。
2. 接管后的故障必须 fail-closed，不得静默回退直连或恢复使用已暂停的账号/号池代理。
3. 订阅和导入文件均视为不可信数据，只提取受支持的出站节点、分组和可安全转换的规则，不执行配置携带的运行时能力。
4. Stone+ 必须独占本地监听器、内部控制器、系统代理租约、TUN 生命周期和 route generation，才能可靠恢复原路由并避免回环。
5. renderer 只接收脱敏摘要和窄 IPC，不接收节点完整凭据、订阅 URL/token 或内部控制器密钥。

因此，Stone+ 与 GUI.for.SingBox 的合理关系是“安全子集加 Stone+ 专用接管协调”，而不是逐字段追求功能对等。

### 3.2 三轴状态事实模型

页面不再把一个 `ready` 字符串或开关值直接翻译成“已接管”，而是交叉验证三条独立事实轴：

1. **用户意图与生命周期轴**：`desiredEnabled` 表示用户目标；`disabled / starting / ready / stopping / error` 表示主进程协调流程所处阶段。用户目标和当前阶段可以暂时不同，例如关闭过程中仍由旧内置代次继续承载请求。
2. **系统接入轴**：`accessState` 单独描述 system/TUN 的 `idle / applying / ready / restoring / error`、接入模式和回环 mixed 端点。Windows 注册表或其他代理软件改写系统代理、TUN 提权被拒绝、恢复失败时，都不能只凭核心存活显示成功。
3. **有效路由轴**：`effectiveRoute` 与 `routeGeneration` 描述 Stone+ 新请求当前实际使用外部路由、内置 mixed/TUN 还是 fail-closed。只有路由类型、代次、mixed 端口、活动配置/节点和接入证据一致时，工作台才发布“已接管”。

候选代次启动或关闭恢复期间，工作台可以明确展示“正在切换，但旧的已发布内置路由仍有效”；这不等于把候选接入标为 ready。核心崩溃或有效路由为 `blocked` 时，则显示 fail-closed，并隐藏可能误导用户的 live mixed 链路。

### 3.3 数据面安全补强

- **节点检索与排序**：先应用已保存的分组过滤，再对 renderer 安全摘要做 NFKC 归一化、多词检索和稳定排序；延迟排序把有效结果、测试中、未测试和失败分层，不会把异常值当成低延迟，也不会原地修改配置节点数组。
- **DNS 边界**：最终配置只接受 1–4 个非回环 IP 上游，默认使用受控的直连 UDP/53 和 IPv4 优先策略；direct 与节点出站的 `domain_resolver` 都引用 Stone+ 自有 DNS tag。订阅不能注入 DNS 监听器、任意 detour、本地文件或脚本。
- **规则边界**：回环直连始终位于规则首部；规则来源明确区分安全转换的配置规则、Stone+ 自定义规则、安全降级、全局和直连。catch-all 只转换为最终动作，固定中国大陆规则集仅在实际需要时加载，并通过选中节点更新以避免直连下载阻塞核心启动。
- **安全 sniff**：针对域名、域名后缀、域名关键词、协议和 geosite 条件，Stone+ 在第一条确需元数据的规则前插入一次 `action: sniff`，超时固定为 `300ms`，且它是继续匹配的非终结动作。纯 IP/端口规则不承担不必要的 sniff 成本；导入的 inbound sniff 选项始终被忽略。
- **连接筛选**：renderer 只过滤脱敏连接摘要，可按目标、来源、网络、协议和规则/出口检索，按目标、TCP/UDP、规则/出口组合筛选，并以连接时间倒序稳定展示。筛选不会扩大控制器权限；断开连接仍使用逐连接 ID 的专用 IPC。

| 原样运行完整 sing-box/Clash 配置           | 通用配置文件编辑与保存                           | 完整透传会让订阅控制 DNS、实验 API、入站、服务、端点和本地资源，破坏配置版本迁移、故障分类、原子切换、回滚以及 fail-closed 不变量。Stone+ 只把受支持字段转换为自己的运行时配置。                                                             |
| 任意本机文件路径                           | 本地规则集、保存路径、证书/密钥路径等            | `script-path`、证书/私钥路径、规则文件路径等可导致核心读取敏感文件，或让配置行为依赖 UI 无法验证的可变外部状态。导入文件只作为一次性内容读取，不成为核心运行时路径。                                                                         |
| 任意用户核心、下载源或版本                 | 内核下载、更新和缓存                             | Stone+ 发布包的完整性、GPL 对应源码和跨平台行为都绑定固定版本及运行目录清单。允许替换核心会使配置兼容、控制 API、TUN 清理和供应链校验不可证明。                                                                                              |
| 任意多入站、多控制器和独立 HTTP/SOCKS 端口 | 完整端口与入站管理                               | Stone+ 的需求是一个可验证的 mixed 出口入口，不是本机代理服务器编排器。增加未纳入租约的监听器会扩大攻击面，并使 LAN consent 和停止清理不完整。                                                                                                |
| 任意接口/路由/DNS 原始参数                 | 完整 TUN、路由、DNS 高级配置                     | 这些字段可绕过节点/DNS/回环排除，造成 PAC → mixed → PAC、TUN 自代理或 DNS 泄漏。Stone+ 只接受经过验证的枚举或预设，并由主进程生成最终配置。                                                                                                  |
| 常驻提权服务或整应用管理员运行             | 管理员模式、后台核心管理                         | Stone+ TUN 每次启动临时提权，用户拒绝时进入明确错误且不回退。常驻高权限会显著扩大 renderer、订阅解析和第三方依赖漏洞的影响范围。                                                                                                             |
| 向 renderer 暴露完整节点/订阅秘密          | 通用前端可直接编辑所有字段                       | renderer 不是凭据保险库。订阅 URL 常含 token，节点字段包含密码、UUID、Reality 密钥等；这些只允许在主进程解密并用于生成短生命周期运行时配置。                                                                                                 |

### “路径”一词的边界

Stone+ 排除的是可指向本机资源或可执行内容的路径，例如脚本、证书、私钥、本地规则集、核心和保存目录。协议握手所需的网络传输字段不属于此类：WebSocket/HTTP Upgrade 的 URL path 和 gRPC service name 可以在对应节点协议受支持时保留，但必须经过长度、字符和结构校验；它们不能被解释成本机文件路径。

## 6. 导入兼容约定

对 GUI.for.SingBox、sing-box 或 Clash Meta 导出的配置，Stone+ 采用以下兼容语义：

1. 兼容是“提取安全交集”，不是配置往返编辑；Stone+ 不保证再次导出后保持原文件结构。
2. 只解析受支持的出站节点、静态分组关系和可完整转换的规则。导入的 inbound、TUN、controller、external UI、endpoint、service、provider、脚本和文件路径不会运行。
3. 单个不支持节点可以被忽略并给出警告；规则只要无法安全维持原语义，就整体切换到可解释的内置规则，避免部分保留造成意外直连。
4. 订阅刷新必须保持稳定节点 ID；活动节点消失时只能选择可见的确定性回退并告知用户，不能悄悄恢复外部代理或直连。
5. 所有生成配置仍需经过 schema 校验、`sing-box check`、端口所有权和健康检查；导入成功不等于接管成功。

## 7. 后续能力的准入条件

任何从“后续候选”进入“已支持”的能力，至少应满足：

- 使用有限、可版本化的数据模型，不接收原始核心 JSON 片段或命令行；
- 在主进程完成格式、范围、地址、大小、超时和凭据校验，renderer 只得到脱敏投影；
- 变更经过串行的两阶段切换，失败可回滚，并且不会提前发布 `ready` 或计划中的 mixed/TUN 地址；
- 不改变用户保存的外部 `direct/system` 设置和账号/号池代理绑定；
- mixed、controller、本地网关、节点和 DNS bootstrap 不进入 PAC/TUN 自代理；
- 覆盖开启、切换、迟到结果、重复点击、崩溃、应用退出、备份恢复和跨平台清理测试；
- 不削弱固定核心版本、运行目录清单、SHA-256 和分发合规要求。
